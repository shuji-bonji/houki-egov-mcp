# 設計ノート

## 設計の3原則

1. **コアは薄く、拡張は厚く** — 将来デジタル庁公式MCPが出ても移行できる
2. **略称辞書を全分野で一本化** — 税法・労働・会計・商事・民事・行政の6分野
3. **bulk DL + FTS5 を差別化価値に** — API方式では出せない横断検索精度

## 想定利用シーン — なぜ6分野を網羅するのか

プロダクト開発・個人事業・フリーランス業務のどこかで、法令調査は必然的に発生する。専門分野を絞った MCP（税だけ、労だけ）では対応しきれない。

| シーン | 必要になる法令例 | 本MCPのカバー |
|---|---|---|
| EC・サブスクサービス開発 | 特商法・景表法・割販法・消契法・個人情報保護法 | ✓ commercial / administrative |
| UGC・SNS サービス開発 | プロバイダ責任制限法・著作権法・個人情報保護法 | ✓ commercial / administrative |
| 決済・暗号資産サービス | 資金決済法・金商法・犯収法 | ✓ commercial |
| ヘルステック・フードテック | 薬機法・食衛法・医師法 | ✓ administrative |
| HRテック・副業マッチング | 派遣法・職安法・労基法・個人情報保護法 | ✓ labor / administrative |
| 不動産テック・建設テック | 宅建業法・建基法・都計法 | ✓ administrative |
| モビリティ・配送 | 道交法・道運法 | ✓ administrative |
| フリーランス青色申告 | 所法・消法・措法 | ✓ tax |
| スタートアップ創業 | 会社法・金商法・独禁法 | ✓ commercial |
| 労務管理・36協定 | 労基法・安衛法・育介法 | ✓ labor |

どのシーンでも「一次情報を引く」「LLM に論点整理させる」「必要なら有資格者に渡す」という使い方は共通。本MCP は**最初の「引く」ステップ**に特化する。

## 既存MCPからの継承マトリクス

| 機能 | 取り込み元 | 改良ポイント |
|---|---|---|
| 略称辞書 | tax-law / labor-law | 全6分野に拡張（約150語） |
| 条/項/号部分取得 | tax-law | API v2 の elm 指定を正しく利用 |
| 目次モード | tax-law | 全ツールに装備 |
| 全文検索 | e-gov-law（思想） | bulk+FTS5 で再実装 |
| 版指定（時点の法令） | e-Gov v2 仕様 | `at=YYYY-MM-DD` |
| Markdown整形 | tax-law / labor-law | デフォルト。JSONはフラグ切替 |
| `legal_analysis_instruction` 的プロンプト注入 | e-gov-law | **排除** |
| 通達・裁決プラグイン | 思想のみ | 独立パッケージ化 |

## API vs Bulk の戦略

| モード | 起動 | オフライン | 全文検索精度 | 初回DL | 用途 |
|---|---|---|---|---|---|
| API（デフォルト） | 即時 | × | API準拠 | 0 | 単発条文引き |
| Bulk（`JP_HOUKI_BULK_CACHE=1`） | 初回遅 | ○ | FTS5 | 数百MB〜 | Skill常用 |

起動時に Bulk モードが有効なら、バックグラウンドで月次アーカイブを取得→SQLite にインデックス。Bulk にある条文は Bulk から即返す。

## 略称辞書のポリシー

- `law_id` は**筆者が確認済みのもののみ**格納。未確認は `null`
- 初回版は6分野で約150語
- 業務使用で必要になった略称を**痛点ログ** (`docs/PAIN-POINTS-TEMPLATE.md`) に記録し、リリース前に反映
- 通称（景表法・PL法・風営法 等）は `aliases` または `note` で補足

## デジタル庁公式MCPとの共存シナリオ

| 公式MCPの内容 | 本MCPの進化 |
|---|---|
| 条文のみ（通達・判例なし） | コア層を公式に委譲、拡張層を独立パッケージ化して継続 |
| 条文＋通達 | 競合しない拡張（裁決・下級裁）のみ残す |
| 薄いまま公開されない | そのまま主役継続 |

どのシナリオでも**拡張レイヤを独立 npm に切り出す設計**なら、無駄な作業はほぼゼロ。

## 開発フェーズ

- **Phase 0**: スケルトン整備（2026-04-23 開始）
- **Phase 1**: e-Gov API v2 コア実装（search_law, get_law, get_toc, resolve_abbreviation）
- **Phase 2**: bulkDL + SQLite FTS5（search_fulltext）
- **Phase 3**: 拡張レイヤ仕様確定、公式拡張パッケージ1〜2本リリース
- **Phase 4**: デジタル庁公式MCPとのアダプタ層（出た場合のみ）

## 開発着手前の必須工程

Phase 1 に入る前に、**2週間の実運用痛点ログ**（`docs/PAIN-POINTS-TEMPLATE.md`）を e-shiwake + tax-law で実施。  
「本当に必要な機能だけ」で MVP スコープを絞ってから着手する。

## 業法との関係

本MCP の立ち位置は「個人利用のセカンドオピニオン・ツール」であり、以下の業法で定める**業務独占**を尊重する設計を採る。

### 関連する業法

| 法 | 条文 | 独占業務の要件 |
|---|---|---|
| 弁護士法 | 第72条 | 有償 × 個別具体の**法律事務**の業としての取扱い |
| 税理士法 | 第52条 | 有償 × **税務代理・税務書類作成・税務相談**の業としての提供 |
| 社会保険労務士法 | 第27条 | **労働社会保険諸法令に基づく申請書等の作成・提出代行**の業としての提供 |

共通して **「有償性」×「個別具体性」×「業としての反復継続」** が揃った場合に発動する。

### 本MCPの設計への反映

1. **判断ロジックを内蔵しない**  
   条文・通達・裁決を取得して返すのみ。「課税か非課税か」「割増賃金はいくらか」といった判断は LLM と利用者の領分。MCP 側は事実に徹する。

2. **一次情報は必ず出典 URL 付きで返却**  
   「AI がそう言った」ではなく「e-Gov のこの URL にこう書かれている」と利用者が確認できるようにする。

3. **利用範囲を DISCLAIMER で明示**  
   業として使う場合は有資格者の監督下でという方針を、README / DISCLAIMER に明記。技術で強制せず、**設計と文書で誘導**する。

4. **拡張レイヤも同じ方針を継承**  
   `ExtensionModule` を書く人も「事実提供に徹する」原則を守ることを推奨（CONTRIBUTING に明記）。

### 「個人が自分の案件に使うこと」は業法の規制外

個人が自分自身の案件を調べるために AI を使うこと自体は、上記の業法の規制対象ではない。有償性が無く、他人の業務として提供しているわけではないため。

ただし、**裁判所・税務署・労働基準監督署は「AI がそう言った」を根拠として受け付けない**。そのため、本MCP は一次情報の URL を返すことを重視している。利用者が必要なら自分で一次情報に飛んで確認できる。

### 権威レイヤとの役割分担

| レイヤ | 提供者 | 性格 |
|---|---|---|
| **権威** | デジタル庁公式 MCP（将来）/ LegalOn / MNTSQ | 「正しさの基準」。有資格者の監修付き、商用ライセンス |
| **セカンドオピニオン** | jp-houki-mcp / kentaroajisaka/tax-law-mcp 等 | ノンベンダー個人 OSS。網羅性・横断性で補完 |
| **判断** | 有資格者（弁護士・税理士・社労士） / 利用者自身 | 最終責任 |

本MCP はセカンドオピニオン層に徹することで、権威レイヤとも判断レイヤとも**競合しない**。それぞれの強みを尊重する。
